# DSL 生成错误分析与应对

## 一、DSL 生成错误的来源

### 错误类型分析

| 错误来源 | 具体表现 | 概率估计 |
|----------|----------|----------|
| **截图识别错误** | 模糊文字误读、列顺序混乱 | 5-10% |
| **标注遗漏** | 漏掉隐藏字段、弹窗内容 | 3-5% |
| **API推断错误** | 组件绑定错误的endpoint | 2-5% |
| **类型推断错误** | input误判为select | 2-3% |
| **级联关系遗漏** | 未识别 dependentOn | 3-5% |
| **DSL语法错误** | 括号不匹配、字段名不合法 | <1%（AI生成相对稳定） |

**总体错误率估计：** 10-20% 的 DSL 可能存在需要人工修正的错误

## 二、错误传播链对比

### 方案A（有DSL）错误传播

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 截图识别错误                                       │
│  如：列名误读为 "fieldNmae"（拼写错误）                       │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 2: DSL 生成                                          │
│  DSL记录: { key: "fieldNmae", title: "字段英文名" }          │
│  ⚠️ 错误被结构化记录，可被发现                                │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 3: 代码生成                                          │
│  代码实现: prop: 'fieldNmae'                                 │
│  ⚠️ 错误传播到代码                                          │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 5: DSL 对比审查                                       │
│  dsl-compare.js 检测：                                       │
│  - DSL: fieldNmae                                           │
│  - YApi: fieldName                                          │
│  → 发现命名不一致                                            │
│  → 报告：NAMING_MISMATCH                                     │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  修复路径                                                    │
│  1. 检查 DSL 源文件                                          │
│  2. 对照 Modao 截图确认正确名称                               │
│  3. 修正 DSL → fieldNmae → fieldName                        │
│  4. 重新生成代码                                             │
│  或：直接修正代码，记录 DSL 需同步更新                         │
└─────────────────────────────────────────────────────────────┘
```

**关键点：** 错误在 DSL 层被记录，审查时可以对比 YApi 发现命名不一致

### 方案B（无DSL）错误传播

```
┌─────────────────────────────────────────────────────────────┐
│  截图识别错误                                                │
│  如：列名误读为 "fieldNmae"                                   │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  直接代码生成                                                │
│  代码实现: prop: 'fieldNmae'                                 │
│  ⚠️ 错误直接进代码，无中间记录                                │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  运行/测试                                                   │
│  API调用失败：                                               │
│  POST /masterDataModel/add                                  │
│  → 后端报错：未知字段 fieldNmae                              │
│  ⚠️ 需要调试才能发现                                         │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  修复路径                                                    │
│  1. 运行报错 → 查 YApi 文档                                  │
│  2. 发现 YApi 定义 fieldName                                 │
│  3. 修正代码                                                 │
│  ⚠️ 不知道其他地方是否还有类似错误                            │
│  ⚠️ 没有系统性检查机制                                        │
└─────────────────────────────────────────────────────────────┘
```

**关键点：** 错误直接进代码，无对比检查，只能靠运行报错发现

## 三、错误发现时机对比

| 错误类型 | 方案A发现时机 | 方案B发现时机 |
|----------|--------------|--------------|
| 字段命名错误 | DSL对比（代码生成前） | API调用失败（运行时） |
| 列顺序错误 | DSL明确定义顺序 | 用户测试发现 |
| 必填字段遗漏 | DSL标注required，代码无验证→检测 | 表单提交被后端拒绝 |
| API绑定错误 | `${api.xxx.list}` 查无此API→发现 | 请求404 |
| 级联遗漏 | DSL无dependentOn→审查质疑 | 用户测试发现无联动 |
| 类型推断错误 | DSL type vs 实际需求→人工review | 用户反馈"不对" |

**核心差异：**
- 方案A：在审查阶段（代码生成前或生成后立即）发现
- 方案B：在运行/测试/用户使用阶段发现

## 四、修复成本对比

### 方案A修复路径

```
发现 DSL 错误 → 修正 DSL → 重生成代码 → 验证
```

**成本：**
- 定位错误：DSL文件明确位置
- 修正时间：1-3分钟
- 重生成：自动化
- 影响范围：清楚哪些代码需要同步修改

### 方案B修复路径

```
运行报错 → 排查原因 → 查文档 → 修正代码 → 不确定是否还有其他错误
```

**成本：**
- 定位错误：需要调试
- 排查时间：10-30分钟/问题
- 影响范围：不确定，可能还有其他类似错误未发现

## 五、DSL 生成错误的风险缓解

### 缓解措施

| 措施 | 说明 | 效果 |
|------|------|------|
| **人工review DSL** | 生成后快速浏览关键字段 | 发现80%识别错误 |
| **YApi字段对比** | DSL字段名 vs YApi响应字段 | 发现命名不一致 |
| **Modao截图对照** | DSL内容与截图快速对比 | 发现遗漏 |
| **分阶段验证** | 先生成1-2页面验证流程 | 提前发现问题 |
| **DSL版本管理** | Git管理DSL，修改可追溯 | 支持回滚 |

### 增加的成本

| 步骤 | 时间 | 说明 |
|------|------|------|
| 人工review DSL | 3-5分钟/页面 | 浏览关键字段 |
| YApi字段对比 | 2分钟 | 检查命名一致性 |
| DSL修正 | 1-3分钟/错误 | 修改并验证 |

**总增加成本：** 5-10分钟/页面（远小于方案B的调试成本）

## 六、客观结论

### DSL生成错误时的情况

| 维度 | 方案A | 方案B |
|------|-------|-------|
| 错误传播 | DSL层记录→可追溯 | 直接进代码→无记录 |
| 发现时机 | 审查阶段（更早） | 运行/测试阶段（更晚） |
| 修复定位 | DSL文件明确位置 | 需要调试排查 |
| 影响范围 | 清楚哪些代码依赖该DSL | 不确定是否有其他类似错误 |
| 系统性检查 | dsl-compare.js 自动对比 | 无自动化检查 |

### 核心差异

**DSL的作用不是消除错误，而是：**

1. **让错误可见** — 结构化记录，便于对比检查
2. **让错误早发现** — 在代码生成前/审查阶段暴露
3. **让错误易定位** — DSL文件明确错误位置
4. **让修复有边界** — 知道哪些代码需要同步修改

### 无DSL的问题

不是错误更少，而是：
1. **错误不可见** — 直到运行报错
2. **发现更晚** — 调试阶段或用户反馈
3. **排查困难** — 需要人工追踪错误来源
4. **修复不确定** — 不知道是否还有其他问题

## 七、实际案例

### 案例：主数据模型字段命名

**DSL生成错误：**
```
DSL记录: { key: "datasourceId", ... }  // 正确 ✅
如果AI误读: { key: "datasourceID", ... }  // ID大小写错误 ❌
```

**方案A处理：**
- dsl-compare.js 对比 YApi：YApi定义 `datasourceId`
- 报告：NAMING_MISMATCH（大小写不一致）
- 修正 DSL → 重新生成代码

**方案B处理：**
- 代码：`datasourceID`
- API调用：参数名不匹配 → 后端可能拒绝或忽略
- 运行报错 → 排查 → 查YApi → 发现命名问题

### 案例：级联选择遗漏

**DSL生成错误：**
```
DSL遗漏 dependentOn:
{ key: "databaseName", type: "select", dataSource: "${api.database.list}" }
// 遗漏: dependentOn: "datasourceId" ❌
```

**方案A处理：**
- 代码生成：databaseName 无级联逻辑
- 审查质疑：为什么 databaseName 没有依赖 datasourceId？
- 补充 DSL → 重生成

**方案B处理：**
- 代码：databaseName 是独立下拉框
- 用户测试：选 datasourceId 后 databaseName 没联动
- 用户反馈 → 排查 → 发现遗漏级联逻辑

## 八、建议

### 降低DSL生成错误的实践

1. **生成后快速review**
   - 对照Modao截图看关键字段是否正确
   - 检查列顺序是否一致
   - 验证级联关系是否标注

2. **自动化对比检查**
   ```bash
   node ~/.claude/skills/pi-review/scripts/dsl-compare.js \
     --dsl-file docs/dsl/frontend/page.dsl \
     --code-dir src/views/module/
   ```
   - DSL字段 vs YApi字段 → 发现命名不一致
   - DSL字段 vs 代码实现 → 发现遗漏

3. **增量验证**
   - 先生成1-2个关键页面
   - 验证流程正确性
   - 再生成剩余页面

4. **DSL版本管理**
   - DSL提交Git
   - 修改有commit记录
   - 便于回滚和追溯

### 最佳实践总结

| 场景 | 建议 |
|------|------|
| 复杂模块 | DSL + 人工review DSL（5分钟） |
| 简单页面 | 可选：DSL或直接生成后加强审查 |
| 高风险字段（级联/必填） | DSL标注 + 验证 |
| 团队项目 | DSL必要（共享文档+版本管理） |

---

## 结论

**DSL生成错误时的核心优势：**

- 不是消除错误（AI识别本身有误差）
- 而是**让错误更早发现、更容易定位、更有边界修复**

**对比无DSL：**
- 无DSL不是错误更少，而是错误发现更晚、修复成本更高
- DSL提供了系统性检查机制，无DSL只能靠运气和人工

**投入产出：**
- DSL生成成本：8-12分钟
- DSL review成本：5-10分钟
- 避免：运行报错调试30分钟-2小时/问题

**结论：DSL + 快速review > 无DSL**
